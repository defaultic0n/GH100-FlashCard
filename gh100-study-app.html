D1: User Support & Workflows
15%
▼
Admins handle config, access, integrations. GitHub Support handles billing, bugs, outages, and GHES installation.
GitHub Premium Support covers: security incidents, GHES installation, outages. Does NOT cover: writing scripts, hardware setup, third-party integrations.
Support priorities: Urgent = critical system failure. High = security business impact. Normal = general questions.
Generate support bundles via ghe-support-bundle over SSH (requires admin SSH access).
GitHub App vs Action: Apps = independent services with webhooks, fine-grained permissions, respond to events across multiple repos. Actions = CI/CD steps inside a single workflow.
Branch protection: require PR reviews, status checks, signed commits, linear history. "Include administrators" forces rules on admins too.
💡 CODEOWNERS file = auto-request specific reviewers for their files. "Dismiss stale approvals" = new commits reset existing approvals.
D2: Identities & Authentication
20%
▼
SAML SSO: Delegates authentication to IdP. Enterprise-wide = single IdP across all orgs (overrides per-org). Per-org = each org can have its own IdP.
SCIM: Automates user provisioning (create), attribute sync (update), deprovisioning (delete). Supported: Azure AD, Okta, custom.
Team Sync: Maps IdP groups → GitHub teams. Requires SAML SSO first. Works with Azure AD and Okta.
SCIM vs Team Sync: SCIM = account lifecycle (create/update/deactivate). Team Sync = team membership only. Both complement each other.
EMU: Accounts created + managed by enterprise IdP. Cannot own personal repos, join outside orgs, modify own profiles.
Non-partner IdP: Requires manual entry of SAML 2.0 details (Sign-on URL, Issuer, X.509 cert). Partner IdPs (Azure AD, Okta) have pre-configured integrations.
2FA: TOTP apps support cloud backup of codes. Members without 2FA are removed when org enforces it.
💡 SAML = Authentication (who?). SCIM = Provisioning (create/update/delete). Team Sync = Team membership. Employee leaves → SCIM auto-deprovisions.
D3: Deployment & Licensing
5%
▼
GHEC: SaaS, GitHub-managed, multi-tenant. SAML SSO, audit log streaming, enterprise policies. Requires internet.
GHES: Self-hosted on your infrastructure. Full data control, you manage upgrades/backups. Best for data sovereignty.
GHAE: Single-tenant, managed by GitHub in isolated cloud environment. Strict compliance. No longer sold to new customers.
GitHub Connect: Links GHES to GitHub.com. Enables GitHub-hosted runners on GHES, license syncing.
Actions pricing: Linux = 1×, Windows = 2×, macOS = 10×. GitHub Discussions engagement is NOT metered.
License = per unique user across all orgs. EMU metered billing = total users including dormant.
💡 GHES = you manage everything. GHEC = GitHub manages. GHAE = GitHub manages in isolated instance.
D4: Access & Permissions
20%
▼
Org Roles: Owner (full control), Member (per defaults), Billing Manager (billing only), Outside Collaborator (specific repos, not org member).
Repo Roles: Read → Triage → Write → Maintain → Admin. Triage = manage issues/PRs (no push). Maintain = settings (no deletion). Admin = full.
Team Maintainer: Add/remove team members, manage nested sub-teams. Cannot change org-wide settings.
Rulesets: Multiple rulesets on same branch = all evaluated, combined rules enforced. Org-level rulesets can target many repos by name pattern.
Custom Repo Roles: Pick specific permissions — e.g., "Manage Settings" without "Delete Repository".
Enterprise policy: Mandatory across all orgs. Org policies operate within what enterprise allows.
💡 Audit token usage: audit log shows token ID + requesting IP + associated user. Audit access: Repo Settings → Manage Access.
D5: Security & Compliance
15%
▼
Secret scanning: Public repos → alerts partner providers (AWS, Stripe) to revoke. Private repos on GHEC → also supports custom pattern definitions.
Push Protection: Blocks a push before it lands if a secret pattern is detected.
CodeQL: Treats code as queryable database with QL language. Finds semantic vulnerabilities, not just patterns.
Dependabot: Dep Graph → Security Alerts → Security Updates (auto-opens PRs). For transitive deps: updates the direct dep to a version that resolves it.
Exposed secrets — order: 1) Revoke/rotate immediately. 2) Clean history (filter-branch/BFG). 3) Contact GitHub Support for cache purge.
Token types: Classic PAT (broad), Fine-grained PAT (scoped to repos), GitHub App Installation Token (short-lived, best for automation), GITHUB_TOKEN (auto, expires after job).
GraphQL Audit Log events: Permission changes, promoting users to admins, GitHub App permission changes. NOT: pushes, clones (those are Git events).
Audit Log Streaming: Real-time delivery to Splunk/Azure Sentinel/SIEM for long-term retention.
💡 GHAS = paid add-on for private repos in Enterprise. FREE for public repos. Includes: code scanning, secret scanning, dependency review.
D6: GitHub Actions
20%
▼
GitHub-hosted runners: Managed, ephemeral, auto-scaled. Standard runners share IPs. Larger runners with static IPs = dedicated, not shared with other tenants.
Self-hosted runners: Custom hardware, private network access. Unique feature: pre/post-job scripts (runner hooks). Risk on public repos: fork PRs can run code on your runner.
Runner groups: Control which repos access specific runners. Configure in runner group settings (not Actions Policies).
Secrets scope: Repository → Environment → Organization → Enterprise. Not passed to fork-triggered workflows by default.
Reusable workflows: Use workflow_call trigger. Called with uses: org/repo/.github/workflows/file.yml@ref.
OIDC: Short-lived tokens for cloud providers (AWS/Azure). Eliminates long-lived secrets.
Jobs without needs run in parallel. concurrency = prevent simultaneous runs.
💡 Squash merge = all commits into one. "Require linear history" = prevents merge commits. Environment secrets = same variable name, different values per env (Production vs Staging).
D7: GitHub Packages
5%
▼
Supported registries: npm, RubyGems, Maven, Gradle, Docker/GHCR, NuGet.
In Actions workflows: use GITHUB_TOKEN to authenticate. From local dev: use a Personal Access Token + username.
Packages vs Releases: Packages = versioned artifacts in a registry (consumed as dependencies). Releases = Git tag + notes + binaries (human-facing). Do NOT use Packages to share code between methods within the same app.
💡 Packages storage and Actions minutes used to build/push packages both count toward billing.
